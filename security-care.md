# セキュリティ上の注意要素

## 1. Supabase（データベース）の脆弱性と対策
最もリスクが高いのがここです。Supabaseはインターネット上にデータベースが公開されるため、設定を間違えると「誰でも他人の打刻データを書き換えられる」「データを全削除される」という状態になります。

### リスク:
悪意のあるユーザーが、アプリの通信を解析してデータベースに直接アクセスし、データを盗み見たり改ざんしたりする。

### 対策（Row Level Security / RLS）:
Supabaseには RLS（行レベルセキュリティ） という強力な防御壁が標準装備されています。「このテーブルは、ログインしている人だけが読み書きできる」「一般アカウントはデータの追加（打刻）だけ、削除は管理者だけ」というルールをデータベース側に設定します。これを必ず有効（Enable）にすることで、不正アクセスを完全に防げます。

## 2. Next.js（フロントエンド）の脆弱性と対策
プログラムのコード内に「見られてはいけない秘密の情報」を書いてしまうミスがよく起こります。

### リスク:
Supabaseの管理者用パスワードや、システム裏側の重要な鍵（APIシークレットキーなど）を、Next.jsの通常のコードにそのまま書いてしまう（ハードコーディング）。Webアプリの画面（ブラウザ）のコードは誰でも「ページのソースを表示」で見られるため、鍵が盗まれます。

### 対策（環境変数 / .env）:
外部に漏れてはいけない重要な鍵は、コード内には直接書かず、「環境変数（.env というファイル）」に隠して管理します。GitHubにもこのファイルは絶対にアップロードしないよう設定（.gitignore に記載）します。

## 3. Vercel（公開サーバー）の脆弱性と対策
Vercel自体は、世界中の大企業も使うほどセキュリティやサーバー攻撃（DDoS攻撃など）への対策が徹底されているため、サーバー自体がハッキングされるリスクは極めて低いです。

### リスク:
一般メンバー用の「共有アカウント」のパスワードが簡単すぎて、研究室外の第三者にログインされてしまう。

### 対策:
研究室のメンバーだけがアクセスできるように、簡単なBasic認証（アクセスした瞬間にパスワードを求められる壁）をかけるか、URL自体を研究室外に漏らさないように運用します。また、今回は個人情報（住所や本名など）は扱わず、「ニックネーム」と「在室ログ」だけを保存する設計にしておけば、万が一の際のリスクを最小限に抑えられます。

## 🛡 開発中に気をつけるべき「鉄則」
私たちが開発を進める上で、徹底すべきことは以下の2点だけです。
- GitHubに「パスワード」や「秘密の鍵」が書かれたファイルを絶対にプッシュ（公開）しない
- Supabaseのデータベースのセキュリティルール（RLS）を必ずオンにする
